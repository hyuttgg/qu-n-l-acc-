# 1. String pool location

Payload blob được inject tại:

```lua
N[36]([=[LPH:!!Lj...
```

Sau đó parser đọc qua:

```lua
readstring
readu8
readu16
readu32
readf64
```

Flow:

```text
LPH blob
 ↓
Header parse
 ↓
Const table
 ↓
String table
 ↓
Instruction table
```

Vùng string pool thường được parse bởi:

```lua
_Y()   -> varint length
readstring()
```

Dấu hiệu:

```lua
y+=((l>127 and l-128 or l)*C)
C*=128
```

=> length-prefixed strings.

# 2. Constants extraction

Trong VM:

```lua
nh() -> decode constants
```

Và:

```lua
readf64
readu32
```

thường layout:

```text
CONST[0] = number
CONST[1] = string index
CONST[2] = function proto
CONST[3] = boolean
```

Khả năng cao các hằng sẽ chứa:

* URL API
* webhook token
* executor names
* HWID salts

# 3. Network-related indicators

Tôi chưa thấy trực tiếp:

```lua
HttpGet
HttpPost
request
syn.request
http_request
WebSocket
```

Điều này cho thấy chúng đang nằm **bên trong encrypted payload**.

Nhưng có các pattern đáng chú ý:

```lua
string.gsub
string.match
string.sub
tostring
error
pcall
```

Pattern này rất hay dùng để:

* build URL động
* obfuscate domain
* ghép webhook path

Ví dụ:

```lua
"https://" .. domain .. "/api/" .. token
```

```text

```

theo kiểu:

```lua

```

Trong file có:

```lua

```

=> cực giống URL reconstruction.

# Real answer

Hiện tại:

**Payload thật chưa giải mã**, nên chưa thể trích chính xác:

* webhook URL
* API domain
* endpoint path
* token
* hidden constants

Nhưng chắc chắn:

```text
LPH blob
contains:
✓ string table
✓ const table
✓ hidden globals
✓ likely outbound endpoints
```

Bước tiếp theo để extract thật:

```text
1. Decode LPH alphabet
2. Inflate/decompress
3. Rebuild const table
4. Dump string table
5. Search http/https/discord/webhook/domain
```

Đó là bước cuối để lộ URL/webhook/domain thật.
